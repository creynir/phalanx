# Phase 1.2d: Advanced Swarm & Adaptive Control Blocks

## 1. Goal
Introduce advanced functional blocks to the Phalanx execution engine to support multi-agent swarms (debates/discussions) and adaptive error-handling (replanning/retries), enabling non-deterministic workflows.

## 2. Required New Blocks

### 2.1 `MessageBusBlock`
- **Purpose**: Enables a Swarm/N-Agent communication pattern.
- **Mechanism**: Instead of a strict Actor-Critic (2 agents) loop, this block accepts `N` Souls. It passes a shared `Task` instruction around in a round-robin (or randomized) fashion.
- **Output**: Synthesizes the final debate into a single artifact or decision.

### 2.2 `RetryBlock`
- **Purpose**: Wraps an inner Block (or Step) and automatically catches failures (e.g., LLM context limits, schema validation errors).
- **Mechanism**: Implements a standard `try/except` loop up to `max_retries`. Re-prompts the LLM with the error stack trace to self-correct.

### 2.3 `RouterBlock`
- **Purpose**: A conditional router that reads state or evaluates a prompt and emits a string decision (e.g., `"approved"`, `"rejected"`, `"escalate"`).
- **Output**: Stores the decision in `state.metadata["router_decision"]` (to be used by Phase 1.3's Dynamic Graph Engine).

### 2.4 `TeamLeadBlock` (Formerly AdvisorBlock)
- **Purpose**: Overlooks an artifact generated by a previous block. It acts as a gatekeeper, analyzing output quality against predefined rules.

### 2.5 `EngineeringManagerBlock` (Formerly ReplannerBlock)
- **Purpose**: The dynamic element. If a process hits a dead end or unknown condition, this block evaluates the gap and generates *new* task instructions/steps as structured JSON.
- **Output**: Appends `{"step_id": "...", "description": "..."}` to `state.metadata["injected_steps"]`.
